# 要件ドキュメント

## 導入
本仕様は、Order という用語を Cart にリネームし、コード全体で一貫した名称を提供することを目的とする。

## 要件

### 1. ドメイン用語の統一
**目的:** 開発者として Cart という用語でドメインを理解し、誤解や認知負荷を減らしたい。

#### 受け入れ条件
1. リネームが完了したとき、システムはドメイン層の公開型・イベント・エラー名として Cart を使用しなければならない。
2. システムは常にドメイン層の公開 API に Order 名の型・イベント・エラーを含めてはならない。

### 2. コマンド/クエリ API の名称統一
**目的:** API 利用者として Cart の語彙で操作でき、読み取り/書き込みの意図が明確になってほしい。

#### 受け入れ条件
1. GraphQL スキーマが生成されたとき、システムは Cart 名のクエリ/ミューテーション/入出力型を提供しなければならない。
2. システムは常に GraphQL スキーマおよび DTO の公開 API で Order 名を使用してはならない。

### 3. モジュール間インタフェースの整合
**目的:** 開発者としてモジュール間の境界を跨ぐ公開インタフェースが Cart 名で統一され、理解しやすくしたい。

#### 受け入れ条件
1. ビルドが実行されたとき、システムはモジュール間で公開される型や関数名に Cart 名を使用しなければならない。
2. システムは常に command/query/rmu/bootstrap の境界を跨ぐ公開インタフェースで Order 名を使用してはならない。

### 4. テスト・ドキュメントの整合
**目的:** メンテナとしてテストとドキュメントが Cart 名に一致し、変更内容を正しく伝えたい。

#### 受け入れ条件
1. テストが実行されたとき、システムは Cart 名で記述されたテストがすべて通過しなければならない。
2. ドキュメントが更新されたとき、システムは Cart 名で統一された説明を提供しなければならない。
